# Implementation Plan: Workspace Reuse & Per-Supernode Allocation Optimization

**Branch**: `021-workspace-reuse` | **Date**: 2026-02-22 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/021-workspace-reuse/spec.md`

## Summary

Eliminate per-supernode heap allocation overhead in the multifrontal LDL^T factorization by pre-allocating reusable workspace buffers sized to the maximum front dimension from symbolic analysis. Currently ~16-18 allocations per supernode (frontal matrix, APTP kernel temporaries, extraction buffers) dominate cost for matrices with many supernodes. The optimization hoists temporary allocations out of the per-supernode loop while preserving per-supernode outputs (FrontFactors, ContributionBlock) as owned allocations.

Two tiers of optimization:
1. **Tier 1 (numeric.rs)**: Pre-allocate frontal matrix buffer and row index buffers, reuse across supernodes
2. **Tier 2 (factor.rs)**: Pre-allocate BLAS-3 inner loop temporaries (backup, L11 copy, LD workspace), reuse across block iterations within and across supernodes

Thread-local workspace via Cell-based move semantics for parallel factorization, matching the existing `g2l` pattern.

## Design Decisions

### D1: Workspace Granularity

**Decision**: Two-level workspace hierarchy.

- `FactorizationWorkspace` (in numeric.rs): Frontal matrix buffer, row indices, delayed columns buffer. One per sequential factorization or one per rayon worker thread. Passed to `factor_single_supernode`.
- `AptpKernelWorkspace` (in factor.rs): BLAS-3 temporaries (backup, l11_copy, LD workspace). Either nested inside FactorizationWorkspace or allocated at the start of `aptp_factor_in_place` from the frontal buffer.

**Rationale**: Separating the two levels keeps the interface clean — numeric.rs doesn't need to know about BLAS-3 block sizes, and factor.rs doesn't need to know about multifrontal orchestration.

### D2: Frontal Matrix Buffer Strategy

**Decision**: Pre-allocate a single `Mat<f64>` of size `max_front x max_front`. Each supernode uses a `submatrix_mut(0, 0, m, m)` view. Zero the lower triangle of the m x m subregion before each supernode.

**Alternative rejected**: Allocate exactly m x m per supernode (current approach). Rejected because this is the primary allocation to eliminate.

**Alternative rejected**: Pool of Mat buffers at different sizes. Adds complexity for minimal benefit over single max-sized buffer.

### D3: Contribution Block Optimization (US2)

**Decision**: Optimize the copy in `extract_contribution` by using column-wise copy via faer's column operations rather than element-by-element. Explore `Mat::submatrix().to_owned()` or similar bulk copy if faer supports it. The contribution block must still own its data (cannot be a view — see research.md R2).

**Full elimination of the copy is deferred**: Restructuring frontal storage into separate fully-summed + trailing regions would eliminate the copy entirely but requires significant refactoring of assembly and factorization paths. This is a post-9.1b optimization if profiling shows contribution extraction is still a bottleneck.

### D4: Thread-Local Workspace Pattern

**Decision**: `thread_local! { static WORKSPACE_BUF: Cell<FactorizationWorkspace> = ... }` in `factor_tree_levelset`, matching the existing `G2L_BUF` pattern.

**Initialization**: Lazily on first use (empty workspace → resize to required dimensions on first supernode). This avoids allocating workspaces for threads that never process supernodes.

**Merge with g2l**: The `global_to_local` buffer currently lives separately (`shared_g2l` in sequential, `G2L_BUF` in parallel). It could be folded into `FactorizationWorkspace`. This simplifies the interface (one workspace parameter instead of two) but changes the existing g2l pattern. **Decision: fold g2l into FactorizationWorkspace** — the patterns are identical and there's no reason to keep them separate.

### D5: Zeroing Strategy

**Decision**: Zero the lower triangle of the m x m subregion before each supernode. Use `fill_zero()` on the MatMut subview or column-wise zeroing.

**Rationale**: Matches current behavior (Mat::zeros allocates and zeros). The workspace buffer is cache-warm from the previous supernode, so zeroing is faster than fresh allocation + OS page zeroing.

### D6: APTP Kernel Workspace Integration

**Decision**: Add workspace buffers for `backup`, `l11_copy`, `ld_workspace`, and `copy_workspace` as a struct passed to `factor_inner` and `two_level_factor`. These are pre-allocated once per `aptp_factor_in_place` call (which is once per supernode) and reused across all block iterations.

**Note**: The per-call outputs (col_order, d, pivot_log) CANNOT be workspace-reused — they are returned as part of `AptpFactorResult` and consumed by `extract_front_factors`. These remain per-supernode allocations.

## Implementation Phases

### Phase A: FactorizationWorkspace & Frontal Buffer Hoisting (US1 core)

**Goal**: Eliminate the largest per-supernode allocations (frontal Mat + row indices).

1. Define `FactorizationWorkspace` struct in numeric.rs with `frontal_data`, `frontal_row_indices`, `delayed_cols_buf`, and `global_to_local` fields
2. Add `FactorizationWorkspace::new(max_front: usize, n: usize)` constructor
3. Add `FactorizationWorkspace::prepare_for_supernode(&mut self, m: usize)` — zeros the m x m lower triangle, clears vectors
4. Modify `factor_single_supernode` to accept `&mut FactorizationWorkspace` and use workspace buffers instead of allocating
5. Modify `factor_subtree_sequential` / sequential path in `factor_tree_levelset` to allocate workspace once and pass to each supernode
6. Run full test suite — verify all tests pass

### Phase B: Thread-Local Workspace for Parallel Path (FR-003)

**Goal**: Extend workspace reuse to parallel factorization.

1. Add `thread_local! { static FACTOR_WORKSPACE: Cell<FactorizationWorkspace> = ... }` in factor_tree_levelset
2. Modify parallel path to take/set workspace from/to thread-local storage (matching g2l pattern)
3. Remove separate `G2L_BUF` — fold into FactorizationWorkspace
4. Run parallel tests — verify correctness under rayon par_iter

### Phase C: APTP Kernel Workspace (Tier 2 optimization)

**Goal**: Eliminate per-block allocations inside factor_inner.

1. Define `AptpKernelWorkspace` struct in factor.rs with backup, l11_temp, ld_workspace, copy_workspace
2. Add constructor from max_front and inner_block_size
3. Modify `factor_inner` to accept `&mut AptpKernelWorkspace` and use workspace buffers for backup, l11_copy, w, l21_copy
4. Modify `apply_and_check`, `update_trailing`, `update_cross_terms` to accept workspace subviews
5. Modify `two_level_factor` to pass workspace through
6. Either allocate AptpKernelWorkspace inside `aptp_factor_in_place` (once per supernode — still an improvement over per-block) or accept from FactorizationWorkspace (zero per-supernode allocations for temporaries)
7. Run full test suite

### Phase D: Contribution Block Copy Optimization (US2)

**Goal**: Reduce contribution block extraction overhead.

1. Replace element-by-element copy in `extract_contribution` (lines 1331-1335) with bulk submatrix copy via faer column operations or `to_owned()` on subview
2. Profile to measure impact — if contribution extraction is no longer a hotspot after Phases A-C, this may be unnecessary
3. Run full test suite

### Phase E: Benchmark Validation & Baseline Update

**Goal**: Verify performance targets are met and no regressions exist.

1. Run full SuiteSparse baseline collection with `diagnostic` feature
2. Compare against post-amalgamation baseline
3. Verify SC-001 through SC-006
4. Update ssids-plan.md and ssids-log.md with results
5. If targets not met, profile to identify remaining hotspots and determine if Phase 9.1c items are needed
