# Implementation Plan: Assembly & Extraction Optimization (Phase 9.1c)

**Branch**: `022-assembly-extraction-opt` | **Date**: 2026-02-22 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/022-assembly-extraction-opt/spec.md`

## Summary

Reduce the 4x performance gap vs SPRAL on c-71/c-big by optimizing the assembly and extraction phases of multifrontal factorization. Post-9.1b profiling shows assembly (37%) and extraction (35%) dominate factor time on these matrices, not the dense APTP kernel (23%). The root cause is O(F^2) per-supernode overhead with tiny elim/front ratios (under 1%).

Three optimization tiers:
1. **Precomputed scatter maps** (US1): Eliminate per-entry index arithmetic in `scatter_original_entries_multi` and `extend_add` by precomputing source→destination mappings at symbolic time.
2. **Bulk column-slice extraction** (US2): Replace element-by-element extraction loops with per-column `copy_from_slice` using faer's `col_as_slice` API.
3. **Bulk frontal zeroing** (US3): Replace element-by-element zeroing with column-slice `fill(0.0)` operations.

## Design Decisions

### D1: Scatter Map Scope — What to Precompute

**Decision**: Precompute two categories of maps during symbolic analysis:

1. **Original-entry scatter map (amap)**: For each supernode, store pairs of `(csc_value_index, frontal_linear_index)` mapping each original matrix entry to its position in the frontal matrix. This follows SPRAL's `amap` pattern.

2. **Extend-add row mapping**: For each child→parent edge in the elimination tree, store a mapping from child contribution row indices to parent frontal row positions. The non-delayed portion is fully precomputable; delayed columns are resolved at factorization time.

**Rationale**: These are the two operations where per-entry index arithmetic dominates. The amap eliminates 5-7 index lookups per nonzero in `scatter_original_entries_multi`. The extend-add mapping eliminates the `global_to_local` lookup per contribution entry in `extend_add`.

**Alternative rejected**: Precomputing only CSC column boundaries per supernode (minimal approach). This saves no per-entry work and only helps with loop setup.

### D2: Frontal Linear Index Encoding

**Decision**: Store destination indices as linear column-major indices into the frontal matrix: `dest = col * stride + row`. The stride equals the front size `m`. At assembly time, the scatter loop becomes `frontal_buf[dest] += a_values[src]` — a single indexed write with no permutation or deduplication logic.

**Constraint**: The linear index depends on the frontal matrix row ordering, which includes the supernode's own columns (fully-summed) and the supernode's pattern (off-diagonal rows). Both are known at symbolic time. However, delayed columns from children insert additional rows at factorization time, shifting positions.

**Resolution**: Compute the amap assuming zero delays (the common case). Store the number of fully-summed columns `k` in the map. At factorization time, if delays occur (rare — 0 delays on c-71/c-big), fall back to the current per-entry assembly path for the affected supernode. This avoids the complexity of parameterized maps while covering the 99%+ case.

### D3: Delayed Column Handling in Extend-Add Maps

**Decision**: The extend-add map stores the mapping from child contribution row indices to parent frontal row positions for the NON-DELAYED case. Specifically, for each child supernode, store a vector of parent-local row indices corresponding to the child's off-diagonal pattern (which forms the contribution block rows when no delays occur).

At factorization time:
- If child has zero delays (common case): use the precomputed map directly.
- If child has delays: the contribution block includes extra delayed rows not in the precomputed map. Fall back to the current `global_to_local` lookup for the affected extend-add.

**Rationale**: Delays are rare (0 on c-71/c-big, 24 on dixmaanl with 60K columns). The precomputed map handles the dominant case. A fallback for delays avoids complex parameterized map designs.

### D4: Memory Layout for Scatter Maps

**Decision**: Store scatter maps as flat `Vec<u32>` arrays with per-supernode offset indices, following CSC-style compressed storage. Use `u32` indices (not `usize`) to reduce memory — frontal matrices are limited to ~11K rows (max_front_size), so 32-bit indices suffice.

Structure:
- `amap_pairs: Vec<u32>` — flattened source/destination pairs for all supernodes
- `amap_offsets: Vec<usize>` — per-supernode start offset into amap_pairs (length: num_supernodes + 1)
- `ea_map: Vec<u32>` — flattened child→parent row mappings for all parent-child edges
- `ea_offsets: Vec<usize>` — per-child start offset (length: total children + 1)
- `ea_child_snode: Vec<usize>` — maps each entry in ea_offsets to its child supernode index (length: total children)

**Memory cost**: ~16 bytes per original matrix entry (two u32 indices per entry) for amap. For c-71 (nnz=859K): ~14 MB. For c-big (nnz=2.3M): ~37 MB. Acceptable per spec clarification (unbounded, proportional to nnz).

### D5: Bulk Extraction Strategy

**Decision**: Use faer's `col_as_slice()` API to access contiguous column data in the frontal matrix, then `copy_from_slice` for each column in extraction loops.

Key constraint: `submatrix_mut` views at row offset > 0 lose column contiguity. Therefore, access the parent `Mat`'s columns via `col_as_slice(j)` and slice into the relevant row range, rather than creating a submatrix view first.

For `extract_contribution` (column j, lower triangle):
```
src = frontal.data.col_as_slice(ne + j)[ne+j .. ne+j+col_len]
dst = data.col_as_slice_mut(j)[j .. j+col_len]
dst.copy_from_slice(src)
```

For `extract_front_factors` L21 (column j):
```
src = frontal.data.col_as_slice(j)[ne .. m]
dst = l21.col_as_slice_mut(j)[0 .. r]
dst.copy_from_slice(src)
```

### D6: Zero Frontal Strategy

**Decision**: Replace the element-by-element zeroing loop in `zero_frontal` with per-column `fill(0.0)` on contiguous column slices. For column j, zero rows j..m (lower triangle) using:
```
frontal.col_as_slice_mut(j)[j..m].fill(0.0)
```

This uses a single `memset`-equivalent per column instead of individual indexed writes.

**Alternative considered**: Lazy zeroing (track which columns are dirty). Rejected due to complexity — tracking dirty columns requires per-supernode bookkeeping and the benefit is marginal since the full lower triangle is typically used.

## Implementation Phases

### Phase A: Bulk Extraction Optimization (US2 — lowest risk, immediate benefit)

**Goal**: Replace element-by-element extraction with per-column bulk copies.

1. Modify `extract_contribution` in numeric.rs to use `col_as_slice` + `copy_from_slice` for lower-triangle column copies
2. Modify `extract_front_factors` L21 extraction to use per-column `copy_from_slice`
3. Modify `extract_front_factors` L11 extraction to use per-column slice copies (where pivot type allows)
4. Run full test suite — verify all tests pass and backward errors are identical
5. Run `profile_matrix` on c-71 — measure extraction percentage change

### Phase B: Bulk Frontal Zeroing (US3 — low risk, incremental benefit)

**Goal**: Replace element-by-element zeroing with per-column fill operations.

1. Modify `zero_frontal` in numeric.rs to use per-column `col_as_slice_mut(j)[j..m].fill(0.0)`
2. Run full test suite — verify all tests pass
3. Run `profile_matrix` on c-71 — measure assembly percentage change

### Phase C: Precomputed Scatter Maps (US1 — highest reward, most complex)

**Goal**: Precompute assembly index mappings at symbolic time.

1. Define scatter map storage structures (per-supernode amap, per-child extend-add map)

3. Implement extend-add map construction — compute child contribution row → parent local row mappings
4. Modify `scatter_original_entries_multi` to use precomputed amap with fallback for delayed supernodes
5. Modify `extend_add` to use precomputed row mapping with fallback for delayed children
6. Run full test suite — verify correctness
7. Run `profile_matrix` on c-71 — measure assembly percentage change
